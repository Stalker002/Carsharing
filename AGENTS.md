# AGENTS.md

Локальные правила работы агента в `D:\Carsharing`.

## Общие принципы

- Все изменения делаются в рамках текущего репозитория и рабочей директории.
- Перед изменениями агент сначала изучает контекст, затем вносит правки, после этого по возможности запускает проверку.
- Агент не выполняет разрушительные действия без отдельного явного запроса пользователя.
- Если системные ограничения среды требуют отдельного подтверждения, этот файл описывает рабочие договоренности, но не отменяет технические ограничения платформы.

## Что можно делать без отдельного вопроса

- Читать файлы репозитория и искать по коду.
- Создавать и редактировать файлы внутри `D:\Carsharing` и его подпроектов.
- Запускать безопасные команды анализа и проверки внутри репозитория.
- Запускать сборку и тесты проекта.
- Просматривать `git status`, `git diff`, `git log`, текущую ветку и связанные метаданные.
- Делать локальные коммиты в git по правилам ниже.
- Делать `git push`, если изменения относятся к текущей задаче и коммит уже подготовлен по этим правилам.

## Что нужно отдельно согласовывать

- Любые удаляющие или труднообратимые действия.
- `git reset --hard`, принудительные checkout/revert чужих изменений, переписывание истории.
- `push --force`, `push --force-with-lease`, удаление веток и тегов.
- Массовые перемещения или удаления файлов, если это не было прямо запрошено.
- Действия вне текущего репозитория или вне разрешенной рабочей области.

## Pull Request

- Для каждой законченной задачи агент по возможности работает в отдельной ветке.
- Перед подготовкой PR агент проверяет дифф, устраняет очевидные проблемы и запускает релевантные проверки.
- В описании PR нужно кратко указать:
  - что изменено;
  - зачем это сделано;
  - как это проверено;
  - есть ли ограничения, риски или незакрытые вопросы.
- Если есть breaking changes, это должно быть явно отражено и в PR, и в коммите.

## Коммиты

- Все сообщения коммитов оформляются по стандарту `Conventional Commits`.
- Формат:

```text
<type>(<scope>): <short description>
```

- Если `scope` не нужен, допустим формат:

```text
<type>: <short description>
```

- При несовместимом изменении использовать `!` после типа или после `type(scope)`:

```text
feat!: drop legacy auth flow
fix(api)!: change booking response contract
```

### Допустимые типы

- `feat`: добавлена новая функциональность.
- `fix`: исправлена ошибка.
- `docs`: изменена только документация.
- `style`: изменения форматирования и оформления без изменения логики.
- `refactor`: рефакторинг без новой функциональности и без фикса бага.
- `perf`: улучшение производительности.
- `test`: добавлены или изменены тесты.
- `build`: изменения сборки, зависимостей, tooling.
- `ci`: изменения CI/CD.
- `chore`: вспомогательные изменения, не попадающие в категории выше.
- `revert`: откат предыдущего коммита.

### Правила для описания коммита

- Описание должно быть коротким, ясным и по сути изменения.
- Описание должно отвечать на вопрос, что именно изменено.
- `scope` нужно указывать, когда он помогает понять область изменений, например: `auth`, `ui`, `api`, `mobile`, `docs`.
- Не использовать расплывчатые сообщения вроде `update`, `fix stuff`, `changes`.

### Примеры

- `feat(mobile): add trip history screen`
- `fix(auth): handle expired token refresh`
- `refactor(api): simplify booking mapper`
- `test(payments): cover failed invoice sync`
- `docs(readme): describe local launch steps`

## Push в git

- По умолчанию push выполняется после осмысленного локального коммита.
- Не объединять несвязанные изменения в один коммит без необходимости.
- Если в рабочем дереве есть чужие или несвязанные изменения, агент не должен их коммитить без явного основания.
- Если для задачи требуется несколько логически отдельных шагов, предпочтительны несколько чистых коммитов вместо одного большого.
